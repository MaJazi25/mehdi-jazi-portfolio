Requirements Analysis and Design

The Requirements Analysis and Design document explains the software needs of Contractor Connect and provides a clear structure for system development. The project is designed as a web application that connects homeowners with local contractors and supports the full process from project posting to bidding, selection, payment, and review.

Purpose and Scope

The main purpose of this document is to describe the high-level software requirements of the system and provide a shared understanding for clients, stakeholders, developers, and instructors. The project scope includes user registration and login, project posting, project browsing and search, bid management, client reviews, payment handling, administrative tools, and the ability to upload and view project documents such as estimates.

System Overview

Contractor Connect is planned as a self-contained web application that improves the traditional process of finding contractors. Instead of relying on word of mouth, social posts, phone calls, or email, the platform brings everything into one place. It focuses on small and medium home projects and provides essential features such as user accounts, project posting, searching, bidding, bid review and selection, dashboards, reviews, and payment support.

Functional Requirements

  • Account creation and login for clients and contractors.
  • Task creation with title, description, category, budget, location, and deadlines.
  • Contractor bidding with cost, time estimates, and qualifications.
  • Client review and selection of bids for project assignment.
  • User dashboards for managing tasks, bids, updates, and notifications.

Design Perspective

The design of the system focuses on usability, clear navigation, and a structured workflow for both homeowners and contractors. Homeowners should be able to post projects and compare bids easily, while contractors should be able to search for projects and submit proposals efficiently. The design also includes an administrator role for moderation, reporting, and system management.

Non-Functional Requirements

The system must be secure, reliable, and easy to use. Non-functional requirements include hashed passwords, role-based access control, HTTPS, server-side validation, protection against injection and XSS issues, and file validation for uploads. The document also sets performance targets such as fast page loading, reliable login, and accurate dashboard data. In addition, the project must work within free hosting limits, use sandbox payments only, and provide local run instructions for demonstrations.

Logical Database Requirements

The system requires a relational database to store user details, project information, bids, payments, and reviews. Data should be stored in structured tables using standard SQL data types and UTF-8 encoding. The database design must support scalable storage, backups, data retention, primary and foreign key relationships, validation rules, and secure access through authentication and authorization controls.
